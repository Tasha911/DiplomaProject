# План автоматизации тестирования

## 1. Перечень автоматизируемых сценариев

**Позитивные UI-сценарии**

Успешная покупка тура по дебетовой карте: заполнение формы валидными данными, отправка, появление сообщения об успешной оплате.

Успешная покупка тура в кредит: заполнение формы кредита валидными данными, отправка, появление сообщения об одобрении кредита.

**Негативные UI-сценарии**

Отказ в покупке по дебетовой карте: заполнение формы невалидными данными, появление сообщения об отказе.

Отказ в выдаче кредита: заполнение формы кредита невалидными данными, появление сообщения об отказе.

***Валидация полей формы (для обеих карт):***
-   Пустое поле «Номер карты»
-   Недостаточность цифр в номере карты
-   Невалидный месяц (00, 13, спецсимволы, буквы)
-   Истекший срок действия карты
-   Пустое поле «Владелец»
-   Невалидное имя владельца (кириллица, цифры, спецсимволы)
-   Невалидный код CVC/CVV (2 цифры, буквы)

**СУБД сценарии**

Запись в СУБД MySQL: проверка создания записей в таблицах. Соответствие статусов дебетовой и кредитной карт и способов оплаты.

Запись в СУБД PostgreSQL: аналогичная проверка сохранения данных для второй базы данных.

Проверка отсутствия сохранения данных карт: валидация того, что номера карт и CVC-коды не попадают ни в одну из таблиц СУБД.

## 2. Перечень используемых инструментов и обоснование выбора

Язык программирования: Java. Промышленный стандарт для автоматизации, строгая типизация, отличная совместимость с тестовыми библиотеками.

Тестовый фреймворк: JUnit 5. Актуальная версия фреймворка. Обладает удобной параметризацией для тестов валидации полей и поддержкой аннотаций жизненного цикла тестов.

Автоматизация UI: Selenide. Удобная обертка над Selenium WebDriver. Имеет автоматические ожидания элементов (умные таймауты), лаконичный синтаксис и встроенные скриншоты при падении тестов.

Работа  с  СУБД: DBUtils (Apache Commons).  Позволяет писать чистые SQL-запросы без перегрузки проекта тяжелыми ORM (вроде Hibernate), что ускоряет проверку состояния БД.

Контейнеризация: Docker и Docker Compose.  Необходимы для локального развертывания официальных образов MySQL, PostgreSQL и запуска Node.js симулятора банковских сервисов (gate-simulator) в единой сети.

Сборщик проектов: Gradle. Более лаконичный синтаксис конфигурации зависимостей по сравнению с Maven, высокая скорость сборки за счет кэширования.

Отчетность: Allure Framework. Позволяет генерировать понятные, детализированные визуальные отчеты со скриншотами UI-шагов для дипломного руководителя.

## 3. Перечень и описание возможных рисков

Проблемы с настройкой окружения в Docker: возможные конфликты портов (8080, 9999, 3306, 5432) на рабочей машине или несовместимость версий Node.js внутри симулятора.

Специфика симулятора (Gate Simulator): симулятор может работать нестабильно, падать под нагрузкой или выдавать не задокументированные ошибки при отправке некорректных JSON-форматов.

Различия в диалектах СУБД: SQL-запросы для проверки данных могут потребовать адаптации из-за разницы синтаксиса или типов данных в MySQL и PostgreSQL.

Плавающие (Flaky) UI-тесты: падения тестов из-за долгой анимации веб-интерфейса или задержек обработки запросов со стороны aqa-shop.jar.

Отсутствие уникальных локаторов элементов: если на веб-странице нет атрибутов data-test-id или id, придется использовать сложные XPath/CSS селекторы, что сделает тесты хрупкими при любом изменении UI.

Разработка тестовой архитектуры (настройка Gradle, подключение библиотек, создание базовых классов): 10 часов.

Написание UI-тестов (позитивные, негативные, валидация форм): 20 часов.

Написание API и DB-тестов (интеграция с MySQL и PostgreSQL, проверка таблиц): 20 часов.

Интеграция отчетности Allure и исправление плавающих тестов (устранение рисков): 10 часов.

Оформление баг-репортов и финального отчета по итогам тестирования: 10 часов.

Итоговая оценка: 80 рабочих часов.
